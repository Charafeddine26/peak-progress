# Peak Progress - Mountain Climbing Tangible Interface

## Project Documentation for M2 IHM TII Tangible Interfaces

**Project Deadline:** March 20, 2026, 23:59

**Target Users:** Patients aged 50+ engaged in adapted physical activity programs

**Core Metaphor:** Mountain climbing as progressive health journey

---

## Executive Summary

**Peak Progress** is a tangible interface that gamifies physical activity engagement through the metaphor of mountain climbing. As users complete their exercise sessions throughout the week, a physical climber figure ascends a 3D-printed mountain, accompanied by progressive LED illumination representing altitude zones. The device combines tactile interaction, visual feedback, and Bluetooth connectivity to create an engaging, non-intimidating way for older adults to track and maintain their physical activity commitment.

**Key Innovation:** Unlike screen-based trackers that can feel clinical or overwhelming, Peak Progress provides a calm, physical representation of progress that exists in the user’s living space as a constant, positive reminder of their health journey. **The progressive mountain unlocking system ensures long-term engagement by introducing increasingly challenging summits that require longer commitment periods.**

## Conceptual Foundation

### Why Mountain Climbing?

The mountain climbing metaphor was chosen for several ergonomic and psychological reasons:

### **1. Universal Symbolism**

- Mountains represent challenge, achievement, and personal growth across cultures
- “Reaching the summit” is an intuitive goal that requires no explanation
- The journey metaphor emphasizes process over outcome

### **2. Progressive Nature**

- Each activity session = one step higher (non-intimidating incremental progress)
- Altitude zones segment the journey into manageable phases
- Multiple mountains create a sense of expedition and adventure
- Longer climbs for harder mountains mirror real mountaineering challenges

### **3. Positive Framing**

- “Climbing” feels adventurous and empowering
- Contrasts with medical/clinical language (“tracking,” “monitoring”)
- Evokes outdoor experiences and nature (appealing to 50+ demographic)
- * “Unlocking new mountains” creates anticipation and achievement**

### **4. Visual Clarity**

- Vertical progress is immediately readable from any angle
- Position = progress (no interpretation required)
- Color-coded zones provide secondary progress indicator
- * Different LED color schemes per mountain create visual variety**

### **5. Reversible Setbacks**

- Missing sessions = slight descent (realistic, not punishing)
- Weekly reset = fresh start (reduces failure anxiety)
- Emphasizes consistency over perfection
- * On multi-week mountains, progress persists across weeks (no complete reset until summit)**

### **Concept: Unlockable Mountain Expedition Chain**

Users begin with a simple training hill and progressively unlock more challenging, iconic mountains. **Each mountain requires an increasingly longer streak to summit**, creating sustained engagement without overwhelming beginners.

### **Mechanical Components**

- **Servo Arm:** Custom 3D-printed extension connected to climber
- **Pulley System:** Small 3D-printed pulley at mountain peak
- **Fishing Line:** Thin cable connecting servo to climber via pulley
- **Vertical Guide Track:** Laser-cut channel ensuring straight vertical movement
- **Climber Figure:** 3D-printed humanoid (~4cm tall) with attachment point

### **Structural Components**

- **Mountain Shell:** 3D-printed in 3-4 interlocking sections (PLA)
- **Base Platform:** Laser-cut acrylic or 3D-printed, houses electronics
- **Translucent Window:** Clear acrylic strip showing internal LED ring
- **Summit Flag:** 3D-printed or laser-cut flag, LED-illuminated, ** swappable for different mountains**

## Software Architecture

### * Data Structures for Mountain System**

```cpp
// MOUNTAIN DEFINITIONS
struct Mountain {
  String name;              // e.g., "Mont Blanc"
  int weeksRequired;        // How many weeks to summit
  int totalSessions;        // e.g., 14 for 2-week mountain
  int tier;                 // 1-4 (Training to Ultimate)
  uint32_t ledColors[10];   // LED color scheme for this mountain
  String summitMelody;      // Audio theme identifier
  bool isUnlocked;          // Has user earned access?
};

// PERSISTENT USER PROGRESS (stored in EEPROM)
struct UserProgress {
  int totalSessionsAllTime;      // Career total
  int currentMountainIndex;      // Which mountain they're on
  int sessionsOnCurrentMountain; // Progress on current climb
  int summitsReached;            // Total mountains completed
  byte unlockedMountains[16];    // Bitfield of unlocked mountains
  unsigned long lastActivityDate; // For streak tracking
};

// MOUNTAIN LIBRARY (defined in code)
Mountain mountains[] = {
  // TIER 1: Training Peaks
  {"Colline Locale", 1, 7, 1, BLUE_TO_GREEN, "SIMPLE", true},
  {"Petit Sommet", 1, 7, 1, BLUE_CYAN_GREEN, "SIMPLE", false},
  {"Mont d'Entraînement", 1, 7, 1, PURPLE_GOLD, "MODERATE", false},

  // TIER 2: Regional Mountains
  {"Mont Blanc", 2, 14, 2, ICE_BLUE_WHITE_GOLD, "TRIUMPHANT", false},
  {"Matterhorn", 2, 14, 2, DARK_PURPLE_RED, "TRIUMPHANT", false},

  // TIER 3: Continental Peaks
  {"Kilimanjaro", 3, 21, 3, WARM_GRADIENT, "EPIC", false},
  {"Denali", 3, 21, 3, ARCTIC_SHIMMER, "EPIC", false},

  // TIER 4: Ultimate Summits
  {"Everest", 4, 28, 4, RAINBOW_SHIMMER, "LEGENDARY", false},
  {"K2", 4, 28, 4, FIRE_PULSE, "LEGENDARY", false}
};
```

### * Arduino Firmware with Mountain Progression**

```cpp
#include<Servo.h>
#include<WiFiNINA.h>
#include<ArduinoBLE.h>
#include<EEPROM.h>

Servo climberServo;
UserProgress userProgress;
Mountain currentMountain;

// INITIALIZATION
void setup() {
  Serial.begin(9600);
  climberServo.attach(9);

  // Load user progress from EEPROM
  loadProgressFromEEPROM();

  // Set current mountain based on saved progress
  currentMountain = mountains[userProgress.currentMountainIndex];

  initializeBLE();
  displayCurrentMountain();
  positionClimberForCurrentProgress();
}

// MAIN LOOP
void loop() {
  // Listen for touch events from Circuit Playground
  if (Serial.available()) {
    char event = Serial.read();
    if (event == 'T') {
      logActivity();
    }
  }

  handleBLECommands();
  checkForNewDay(); // Streak tracking
}

//  CORE MOUNTAIN PROGRESSION LOGIC
void logActivity() {
  userProgress.sessionsOnCurrentMountain++;
  userProgress.totalSessionsAllTime++;
  userProgress.lastActivityDate = getCurrentDate();

  // Calculate position on current mountain
  float progressPercent = (float)userProgress.sessionsOnCurrentMountain /
                          currentMountain.totalSessions;
  int servoPosition = 20 + (progressPercent * 150); // 20° to 170°

  // Move climber
  moveClimber(servoPosition);

  // Update LEDs based on progress and mountain color scheme
  updateLEDsForProgress(progressPercent);

  // Play audio feedback
  playProgressSound();

  // Save progress
  saveProgressToEEPROM();

  // Check if summit reached
  if (userProgress.sessionsOnCurrentMountain >= currentMountain.totalSessions) {
    summitReached();
  }

  // Notify mobile app
  notifyMobileApp();
}

void summitReached() {
  // Celebration!
  celebrateSummit(currentMountain.name);

  // Increment summits counter
  userProgress.summitsReached++;

  // Check what mountains to unlock
  unlockNextMountains();

  // Move to next mountain in progression
  if (hasUnlockedMountains()) {
    currentMountain = selectNextMountain();
    userProgress.currentMountainIndex = getCurrentMountainIndex();
    userProgress.sessionsOnCurrentMountain = 0;

    // Announce new mountain
    announceNewMountain(currentMountain.name);
  }

  // Save progress
  saveProgressToEEPROM();

  // Reset climber to base for new mountain
  descendToBase();
}

//  UNLOCK LOGIC
void unlockNextMountains() {
  int summits = userProgress.summitsReached;

  // Tier 1 unlocks
  if (summits >= 1) unlockMountain("Petit Sommet");
  if (summits >= 2) unlockMountain("Mont d'Entraînement");

  // Tier 2 unlocks
  if (summits >= 3) unlockMountain("Mont Blanc");
  if (summits >= 4) unlockMountain("Matterhorn");

  // Tier 3 unlocks
  if (summits >= 5) unlockMountain("Kilimanjaro");
  if (summits >= 6) unlockMountain("Denali");

  // Tier 4 unlocks
  if (summits >= 7) unlockMountain("Everest");
  if (summits >= 8) unlockMountain("K2");
}

void unlockMountain(String mountainName) {
  for (int i = 0; i < sizeof(mountains)/sizeof(Mountain); i++) {
    if (mountains[i].name == mountainName) {
      mountains[i].isUnlocked = true;
      // Set bit in EEPROM bitfield
      bitSet(userProgress.unlockedMountains[i/8], i%8);

      // Notify user
      Serial.println("UNLOCK:" + mountainName);
      BLE.write("NEW_MOUNTAIN:" + mountainName);
    }
  }
}

//  LED UPDATES FOR MULTI-WEEK MOUNTAINS
void updateLEDsForProgress(float percent) {
  int ledsToLight = (int)(percent * 10); // 0-10 LEDs

  // Send color scheme to Circuit Playground
  for (int i = 0; i <= ledsToLight; i++) {
    uint32_t color = currentMountain.ledColors[i];
    Serial.print("LED:");
    Serial.print(i);
    Serial.print(":");
    Serial.println(color, HEX);
  }
}

//  HANDLING MULTI-WEEK MOUNTAINS
void checkForNewDay() {
  unsigned long today = getCurrentDate();

  if (today != userProgress.lastActivityDate) {
    // New day started

    // For multi-week mountains, check if it's end of week
    int dayOfWeek = getDayOfWeek(today);

    if (dayOfWeek == 1) { // Monday
      // Check if we completed this week's portion
      int weekNumber = (userProgress.sessionsOnCurrentMountain / 7) + 1;
      int sessionsExpectedThisWeek = weekNumber * 7;

      if (userProgress.sessionsOnCurrentMountain >= sessionsExpectedThisWeek) {
        // Week completed! But mountain continues
        celebrateWeeklyMilestone(weekNumber);
        // Climber STAYS at current position
      } else {
        // Missed some days, gentle slide down
        handleMissedDays();
      }
    }
  }
}

void celebrateWeeklyMilestone(int weekNum) {
  Serial.println("WEEKLY_MILESTONE:" + String(weekNum));

  // Special animation but not full summit celebration
  Serial.println("LED:WEEK_COMPLETE");
  playTone(C5, 300);
  playTone(G5, 500);

  // Notify app
  BLE.write("WEEK_" + String(weekNum) + "_OF_" +
            String(currentMountain.weeksRequired) + "_COMPLETE");
}

//  PERSISTENT STORAGE
void saveProgressToEEPROM() {
  int addr = 0;
  EEPROM.put(addr, userProgress);
}

void loadProgressFromEEPROM() {
  int addr = 0;
  EEPROM.get(addr, userProgress);

  // Restore mountain unlock states
  for (int i = 0; i < sizeof(mountains)/sizeof(Mountain); i++) {
    mountains[i].isUnlocked = bitRead(userProgress.unlockedMountains[i/8], i%8);
  }
}
```

### * Circuit Playground Firmware Updates**

```cpp
#include<Adafruit_CircuitPlayground.h>

// Color schemes for different mountains
const uint32_t BLUE_TO_GREEN[10] = {
  0x0000FF, 0x0033FF, 0x0066FF, 0x0099FF, 0x00CCFF,
  0x00FF99, 0x00FF66, 0x00FF33, 0x00FF00, 0x33FF00
};

const uint32_t ICE_BLUE_WHITE_GOLD[10] = {
  0x00FFFF, 0x33FFFF, 0x66FFFF, 0x99FFFF, 0xCCFFFF,
  0xFFFFFF, 0xFFFFCC, 0xFFFF99, 0xFFDD44, 0xFFAA00
};

const uint32_t RAINBOW_SHIMMER[10] = {
  0xFF0000, 0xFF7F00, 0xFFFF00, 0x00FF00, 0x0000FF,
  0x4B0082, 0x9400D3, 0xFF00FF, 0xFF66FF, 0xFFFFFF
};

void setup() {
  CircuitPlayground.begin();
  Serial.begin(9600);
}

void loop() {
  // Touch detection
  if (CircuitPlayground.readCap(3) > 500) {
    Serial.write('T');
    delay(500);
  }

  // Listen for LED commands from Arduino
  if (Serial.available()) {
    String command = Serial.readStringUntil(':');

    if (command == "LED") {
      int ledIndex = Serial.parseInt();
      Serial.read(); // Skip ':'
      uint32_t color = Serial.parseInt();
      CircuitPlayground.setPixelColor(ledIndex, color);
    }
    else if (command == "LED" && Serial.peek() == 'W') {
      // Weekly milestone animation
      weeklyAnimation();
    }
    else if (command == "LED" && Serial.peek() == 'S') {
      // Summit celebration animation
      summitAnimation();
    }
  }
}

void weeklyAnimation() {
  // Gentle pulse
  for (int brightness = 0; brightness < 255; brightness += 5) {
    for (int i = 0; i < 10; i++) {
      CircuitPlayground.setPixelColor(i,
        CircuitPlayground.gamma32(CircuitPlayground.ColorHSV(0, 0, brightness)));
    }
    delay(10);
  }
}

void summitAnimation() {
  // Rainbow cascade
  for (int cycle = 0; cycle < 3; cycle++) {
    for (int i = 0; i < 10; i++) {
      CircuitPlayground.setPixelColor(i, CircuitPlayground.colorWheel(i * 25));
      delay(50);
    }
  }
}
```

### For Technical Report

### **A. Ideal Interface (Ergonomic & HCI Perspective)**

*Discussing constraints for 50+ demographic:*
- Large, tactile interaction area (no small buttons)
- High-contrast visual feedback (bright LEDs, clear position)
- Auditory confirmation (for visually impaired users)
- Meaningful physical feedback (climber movement = tangible progress)
- Calm aesthetic (not overwhelming, fits home environment)
- No screen dependency (respects digital divide)
- ** Progressive challenge system prevents boredom without overwhelming beginners
**-**  Multi-week mountains build sustained habits (research shows 21+ days for habit formation)**

### **B. Justification of Metaphors Used**

### **Primary Metaphor: Mountain Climbing**

*Why climbing metaphor succeeds:*
- Universally understood symbol of achievement across cultures
- Progressive journey (not binary success/failure)
- Ties physical activity to adventure (positive emotional reframing)
- ** Multi-week climbs mirror real mountaineering expeditions
**-**  Unlocking famous peaks creates aspirational goals**
- Altitude zones provide intermediate psychological waypoints

### * Secondary Metaphor: Expedition Progression**

*Why unlockable mountains enhance engagement:*
- **Real-world parallel:** Mountaineers build skills on smaller peaks before attempting Everest
- **Mastery progression:** Clear skill curve from beginner to expert
- **Intrinsic motivation:** Unlocking is reward itself, not external prize
- **Sustainable challenge:** Prevents plateau effect common in fitness tracking
- **Narrative arc:** Users tell story: “I’ve climbed from local hills to Kilimanjaro”

### **Metaphor Validation for 50+ Demographic:**

- **Outdoor nostalgia:** This generation often has strong positive associations with nature/hiking
- **Achievement orientation:** Values long-term goals over instant gratification
- **Storytelling:** Enjoys sharing journey narratives with family
- **Non-digital native friendly:** Physical mountain more intuitive than abstract “levels”

### **C. Prototype Design Justifications**

*Addressing constraints with mountain progression system:*

| Constraint | Design Decision | Justification |
| --- | --- | --- |
| **Material** | Single physical mountain with LED color changes | Ecological: Reuses same hardware, variety comes from software |
| **Size** | 25cm height accommodates all mountains | Optimal visibility without space intrusion |
| **Power** | Low-power LEDs + servo | EEPROM persists data through power cycles; minimal energy use |
| **Age-appropriate** | 1-week starter mountains | Builds confidence before multi-week commitment |
| **Retention** | Progressive unlock system | Combat abandonment (common in fitness devices after 2-3 weeks) |
| **Coding** | Modular mountain struct array | Easy to add new mountains post-launch |
| **Time constraint** | Core system in 2 weeks, mountain library expandable later | MVP with extensibility |

**Key Algorithm: Multi-Week Progress Calculation**

```
Given:
- Mountain M requires W weeks (e.g., Mont Blanc = 2 weeks)
- Total sessions S = 7 * W (e.g., 14 sessions)
- Current session count C

Calculate:
- Overall progress: P = C / S
- Servo position: angle = 20 + (P * 150)
- LED count: lights = floor(P * 10)
- Week number: week = floor(C / 7) + 1

Handle edge cases:
- Missing days → gentle descent (max 1 position per 2 days missed)
- Weekly milestone → celebrate but don't reset climber
- Summit reached → unlock next tier
```

### **F. Difficulties & Solutions**

| Challenge | Solution | Implementation |
| --- | --- | --- |
| **Servo jitter at endpoints** | Software limits: 20°-170° instead of full 0°-180° | `climberServo.write(constrain(angle, 20, 170))` |
| **Fishing line slipping** | Textured servo horn + knot with hot glue seal | Physical mod to horn, tension adjustment |
| **LED visibility through shell** | Translucent window + internal light diffuser | Laser-cut frosted acrylic strip |
| **Bluetooth range issues** | Antenna positioning tests, moved Arduino to base | Avoid metal interference, keep antenna exposed |
| **Capacitive touch false triggers** | Debounce delay + threshold calibration | 500ms delay, threshold tuning per environment |
| ** EEPROM wear from frequent writes** | Write only on state changes, not every loop | Reduces writes from ~1M/day to ~10/day |
| ** Power loss during multi-week climb** | EEPROM saves progress immediately after each session | User never loses more than 1 session data |
| ** Color scheme variety with 10 LEDs** | Use intermediate brightness + hue gradients | `ColorHSV()` function creates smooth transitions |
| ** User confusion on multi-week mountains** | OLED displays “Week X/Y”, app shows detailed progress | Clear visual indicators at all times |
